Digit User Manual¶

Preface¶

This manual contains instructions for setting up and safely operating the robot. It is divided into two primary components:

  • Hardware: This section includes information on safety, system specifications, initial unboxing, transportation, operation, and service.

  • Software: This section includes information on configuring the system, interfacing with the robot software, developing user applications, and simulating environments.

Important

Additional details on using this manual should be read before proceeding. All users should read all the provided safety information and be familiar with the system before attempting to operate the robot.

Using This Manual¶

Emergency Stop¶

Each robot ships with a FORT Robotics Wireless Emergency Stop. This transmitter is required to be powered on and released for the robot to perform any locomotion or manipulation tasks. In the event of an emergency, the transmitter will disable all motors on the robot within 1 second of pressing the button.

Important

To disable the robot in an emergency situation, firmly press the large red button on the E-Stop transmitter until it clicks in place.

Caution

When the robot is disabled in this way, it will have no means to slow its fall or avoid obstacles/people. Depending on the environment this could cause the robot to fall on people or other equipment, resulting in potential injury or damage. Only use the E-Stop in emergency situations when the robot is performing hazardous behaviors and is not responding to other commands.

To turn on the E-Stop, make sure the large red button is depressed, and press the small black button by the antenna on the bottom of the transmitter. The LED should blink slowly while the transmitter is on. The LED on the red button should be a solid red to indicate the robot is disabled. To enable the robot, twist the red button counter-clockwise and it will unlock and pop up. At this point the red light will start to blink slowly, indicating a connection to the receiver in the robot.

When the transmitter battery is low, the small black power button will blink quickly. At this point you should charge the transmitter using the included charger (it will connect to the mini-USB port on the bottom of the transmitter). The E-Stop transmitter will continue to operate while charging, so you may continue to use the robot as long as the E-Stop receiver remains within easy reach of the robot operator

Important

The transmitter must be powered on to charge.

Basic Assembly¶

  1. Open box #1 and remove the branded mat. Lay it on the ground in the center of your work space.

  2. Take a moment to familiarize yourself with the contents of the torso case. Set aside the toolkit to use later.

  3. Take a moment to familiarize yourself with the contents of the leg case. Set aside the screws to use later.

  4. Grab the torso by the exposed handle and place it onto one side of the mat so that the shoulders and LiDAR are close to the top. Avoid touching or grabbing the LiDAR.

  5. Open box #2 and remove one of the legs, unfold it and place it onto the mat in its respective location, as shown below.

  6. Mate the leg to the torso by lining up the following alignment features in the order specified. After both pins are aligned, press the leg into the core.

  7. Remove 4 x M6-16 nylon patch screws from the bag labeled “limb screws”, and tighten the two at the bottom of the torso lightly by hand. Then install and tighten the two on the side of the torso. Torx keys are included in the toolkit.

  8. Repeat the leg alignment and screw installation steps for the 2nd leg.

  9. Remove the torque wrench from the toolkit, assemble it as shown, and torque the 4 screws on the bottom of the torso until a tactile click is felt / heard. Then proceed to torque the 4 screws on the side of the torso. The torque spec is 21 N-m; it will feel like a lot of force.

Battery Installation¶

Important

The Battery Safety instructions must be read and understood before proceeding.

  1. Remove the rear shell from Digit by removing 4 button head screws, set the screws aside.

  2. Remove the battery from its transportation case and align it with the 6 pins on the torso. Make sure that the battery is clocked properly so that the power connector lines up with the mating connector on the torso. Gently press the battery onto the torso to seat the connector.

  3. Secure the battery with the labeled battery screws and washers. Tighten these bolts by hand with the included allen keys.

  4. Install the payload shield onto Digit and fasten with the included button head screws.

  5. Reinstall the rear shell with the included button head screws. Torque by hand until snug.

Service¶

This section details the procedures for preventative inspections and the replacement of wear items on Digit.

Important

Aside from the instructions detailed here, Digit should not be disassembled without explicit instruction from Agility Robotics Inc.

Service Schedule¶

To best maintain the functionality and durability of Digit under normal use, Agility Robotics recommends that you perform regular inspections on Digit according to the service intervals listed below. These inspections are designed to ensure than any problems that arise in your Digit robot are caught quickly, and before they escalate.

Each Day¶

  • While the Digit is powered off, move the hands and feet of Digit to back-drive all actuators. If you feel a large change in backlash, stiction or cogging from one day to the next please contact our support desk.

  • Ensure that the connector seals are properly seated.

  • Inspect all of the optical surfaces for signs of smudging / damage. If the lenses are dirty you should clean them with a lint free alcohol wipe.

    • There are 3 optical surfaces towards the top of the torso; the LiDAR, the context camera and the RealSense camera.

    • There are 3 more optical surfaces on the bottom of the torso.

Each Week¶

  • Check the tightness of the limb mounting screws. If any screws are loose they should be re-torqued.

  • Inspect both the tarsus and achilles connecting rods for any slop in the rod-ends. There are 6 rod ends to check per leg. Limbs that have rod-ends with excessive backlash should be returned to Agility Support for maintenance.

  • Inspect the shells for signs of wear and for damage to the foam. Check tightness of all mounting screws.

  • Inspect the foot soles for signs of wear and check the mounting screws for tightness.

  • Inspect all of the cooling fans to verify airflow.

    • There are two CPU cooling fans on the torso.

    • There are two cooling fans on each thigh; one on the top and one on the bottom.

    • There is a cooling fan in each bicep.

    • There is one cooling fan on each of the 4 actuators that bolt to the torso.

IMU Calibration¶

The magnetometer, gyroscope and accelerometer sensors in the IMU can drift over time and develop biases that can impact the performance of Digit’s state estimation. If you notice a deterioration in the walking stability of Digit (most commonly manifesting as a significant drift to either side during walking) you may need to recalibrate these sensors by following these steps:

  • Turn the robot off and then back on.

  • Once the web interface starts, command the robot to stand up.

  • Send a action-calibrate-imu message.

Battery Storage¶

Important

The battery should be stored with a charge level between 25% and 40%. Please note the following recommendations:

  • Prior to placing the battery in storage, charge to a level of approximately 40%.

  • A spare battery that is not installed in a robot may be stored for up to 60 days when properly charged.

  • Batteries should be stored at room temperature or slightly cooler (15 C to 22 C).

  • If the battery is not properly charged before storage and/or left unattended for an extended period of time, the charge level may drop to zero, causing internal safety features to disable the battery circuitry. Permanent damage to the battery is possible below zero state of charge. If this occurs, you should contact Agility Support for assistance.

When connected to a robot, the battery state of charge should be checked weekly and maintained above 30% when not in use. When the robot is being stored/transported, the battery should be removed from the robot and maintained as described above.

Software¶

Conventions¶

In this chapter we provide general documentation about major frames and their conventions used to represent the robot in a physical or simulated world. Some familiarity with rigid body dynamics is assumed in this chapter. Wikipedia has a good introduction, and a detailed reference is also linked here.

Units of Measurement¶

Unless stated otherwise, SI base units (i.e. meters, kilograms, seconds) are used throughout the robot software for user input, feedback, and log data, with the exception that temperatures are specified in Celsius rather than Kelvin. Angles are generally specified in radians, though in some cases it is possible to explicitly specify values in degrees.

Coordinate Systems¶

Unless specified otherwise, all coordinate systems used by the robot are right-handed Cartesian frames, with the X axis pointing forward, Y pointing left, and Z pointing up. Rotations are defined using the right-hand rule (i.e. counterclockwise) around each of these axes.

Rotation Conventions¶

Whenever angles are used to specify 3D rotations, they are given as [roll, pitch, yaw], where the definitions of each angle follow the common definitions used for aircraft. This convention can also be referred to as extrinsic XYZ Euler angles, and is equivalent to intrinsic ZYX Euler angles with the components listed in reverse order. Radians are used instead of degrees unless explicitly specified otherwise.

Unit quaternions used to specify 3D rotations are given as 4-element vectors [w, x, y, z] such that q = w + ix + jy + kz and i^2 = j^2 = k^2 = ijk = -1.

Kinematic Tree¶

The robot is modeled as an articulated rigid body tree with a floating base. The image below depicts all the body frames (X-red, Y-green, and Z-blue). The robot’s torso is used as the root of the tree. The body frame of the torso has the X axis pointing forwards, the Y axis pointing to the left, and the Z axis pointing up when the robot is standing upright. The frames of all other bodies have their Z axis along the rotation axis of the joint connecting it to the previous body in the tree pointing out from the output side of the joint. Each joint in the tree is revolute joint with the positive direction defined as counter-clockwise around the Z axis of the body moving around the joint.

AprilTags¶

AprilTag is a type of machine-readable fiducial tag used to tag physical objects and locations. Digit is configured to recognize a specific set of AprilTags and can estimate their pose relative to the robot. These tags can be printed on normal printer paper and used for object detection, map localization, and as reference frames for commands.

Tags recognized by the robot are assigned a frame with the origin at the center of the tag and the X axis pointing outwards, such that when the tag is placed on a wall, +X values are in the space in front of the wall and -X values are inside the wall. Tags have an unambiguous orientation, with the Z axis pointing in the upwards direction and the Y axis pointing to the right from the point of view of one facing the tag. In order to be recognized correctly, tags must be square.

Using Tags¶

AprilTags are objects that can either be used directly as a reference frame or for inferring the pose of other objects like boxes or map frames. When the robot detects an AprilTag it will create an object if it doesn’t already exist and update the pose of that object using the measurement directly. Examples of AprilTags being used for box tracking and localization can be found in the examples.

Refer here to set the parameter which overrides the default tag size value. The detection range depends on the size of the tag in the camera frame, increasing the tag size increases the detection range but at the same time it also increases the minimum distance to detect the tags. The tags need to be completely visible in the camera frame to be detected. A tag of size 162 mm has detection range of approximately 3 m when in the FOV of a RealSense D435 color stream with 720x1280 resolution.

Example: perception.april-tag.tag-size-overrides = [{id=1, size=0.15}, {id=2, size=0.1}].

Printing Tags¶

Digit is configured to recognize AprilTags, which are provided as images that can be printed with an ordinary office printer. Any of the tags can be used for a given task, but users should take care to prevent duplicate tags from being used in a way that will confuse the robot. The robot will only remember the most recent observed tag if multiple duplicate tags are used.

  1. Download and extract the zip archive.

  2. Select one or more tags to print. Each tag is identified by a number in the filename, which is the same as the ID number used to specify a tag in commands and other messages.

  3. Determine the printer settings that result in a correctly sized tag. This will vary between paper types, operating systems, and printers.

    • The default tag size is set to 162 mm when measured along each edge of the black square. Each tag image is 750x750 pixels including 75 pixels of white border on each side.

    • To print a tag of specific size, calculate and use the image size or pixel/mm ratio to print the tag. Example: An image of size 125 mm will have 10%(12.5 mm) white border in each side making the tag size 100 mm and pixel/mm ratio of 750/125 that is 6.0 pixels/mm.

  4. Print out the tags using the appropriate settings and verify their size. Note that the tag ID number is present in dark gray in the bottom left corner of the black square, though it may not be visible with some printer types and settings.

    • If the number is not readily visible, we recommend using a pen to note the ID number and orientation of the tag, while taking care to leave appropriate blank whitespace (10% of the image size) around the black square.

Tip

It is recommended to print tags on heavy weight paper to increase durability and prevent wrinkles.

Networking¶

This document describes the network configuration on the physical Digit robot.

Important

To maintain security, Digit’s network interfaces should never be made directly accessible from any untrusted network, especially the public internet. If remote network access to the robot is required for an application, it should be done through a secure link (such as a VPN) to a customer-provided computer installed in the payload bay.

Network Interfaces¶
Payload Ethernet¶

The payload ethernet port on Digit is configured with the IPv4 address and subnet 10.10.1.1/24. Additionally, a DHCP server is running on this port, which will automatically assign addresses between 10.10.1.100 and 10.10.1.199 to DHCP clients. All network services, including Secure Shell and the JSON API, are accessible over this network link.

The recommended way to access the payload ethernet interface is to connect a cable directly between the port on the robot and an ethernet port on another computer. The other computer can either be configured as a DHCP client, in which case it will be assigned an IP address that allows it to communicate with Digit as 10.10.1.1, or it can be assigned a static IP address in the range of 10.10.1.2 to 10.10.1.99 with a netmask of 255.255.255.0.

Note

Devices connected to the payload ethernet interface are able to communicate with devices on the wireless network. The correct routing for this should be automatically configured when using DHCP, but if a static IP is used instead then the client will have to be configured to route requests to the 10.10.2.0/24 network through the 10.10.1.1 gateway.

Important

Digit’s payload ethernet port should not be directly connected to a typical home or corporate network. Digit acts as a DHCP server, and not a DHCP client, and this will potentially interfere with other devices on the network.

Wireless Network¶

Digit hosts a wireless network that can be used to access all network services. The SSID will consist of the robot’s serial number by default, and a unique default password is distributed with each robot. The wireless network is configured with the IPv4 address and subnet 10.10.2.1/24. Additionally, a DHCP server is running on this port, which will automatically assign addresses between 10.10.2.100 and 10.10.2.199 to DHCP clients. All network services, including Secure Shell and the JSON API, are accessible over this network link.

The recommended way to access the wireless network is to connect to it as you normally would with a laptop, smartphone, or other WiFi client device, relying on DHCP to configure the network settings. The device will not get internet access from the Digit WiFi connection, but it will be possible to access the robot’s web interface by opening http://10.10.1.1:8080/gamepad/ in a browser.

Note

Devices connected to the wireless network are able to communicate with devices connected to the payload ethernet. The correct routing for this should be automatically configured when using DHCP, but if a static IP is used instead then the client will have to be configured to route requests to the 10.10.1.0/24 network through the 10.10.2.1 gateway.

Modifying the Wireless Network Settings¶

The wireless network can have its SSID and password changed, or be disabled entirely, by using the Digit Settings interface at http://10.10.1.1/settings/. Robot settings have the following restrictions:

  1. WiFi password must be between 8 and 64 characters long and consist of only valid printable ASCII characters (including spaces)

  2. The name must consist of only valid printable ASCII characters (including spaces), and must be shorter than 32 characters

Note

If something goes wrong with configuring the WiFi settings on Digit and you can no longer access the wireless network, you should still be able to access the payload ethernet. You can use that to reset the WiFi password and other settings.

Services¶

The robot’s computer provides several services that can be accessed over either of the network interfaces. The following is not an exhaustive list, but instead describes the services that are available only on the physical robot, as opposed to those that are also available with the simulator.

Secure Shell¶

Secure Shell (SSH) provides a way to access a remote terminal session on the robot’s computer and copy files to and from the robot. Password authentication is disabled, but a unique SSH key is distributed with each robot allowing access using the root user account.

Assuming a unix-like system with OpenSSH installed, you can specify the key to use with each ssh or scp command using the -i option (e.g. ssh -i digit_ssh_key root@10.10.1.1). To avoid having to specify the key in every command, put the key in ~/.ssh/digit_ssh_key add the following to ~/.ssh/config (create this file if it does not exist):

Host digit
    HostName 10.10.1.1
    User root
    IdentityFile ~/.ssh/digit_ssh_key

After the above setup, you should be able to log into the robot by running ssh digit. The following instructions show an example of how to log in and copy files:

  1. Set up the files in the ~/.ssh/ directory as described above.

  2. Connect to one of the robot’s network interfaces.

  3. Run the command scp digit:/agility/readonly/robot_name ./ to copy the robot’s default name (i.e. its serial number) to the current directory on your local system. You can inspect the contents of this file by running cat robot_name.

  4. Run the command ssh digit to start a remote terminal session on the robot’s operating system.

  5. Run cat /agility/readonly/robot_name in the remote terminal session to print out the contents of the default robot name file on the robot’s filesystem, which should match the file that was copied earlier.

  6. Run exit in the remote terminal session to quit and get back to the local system terminal.

Important

Do not make any modifications to the robot’s filesystem except as described in this documentation or as directed by Agility support. Doing so could render the robot unbootable or cause issues with the robot’s control software.

Additional SSH keys can be added by creating a file at /agility/settings/authorized_keys. This file should be in the standard OpenSSH authorized_keys format, with one public key per line, and must have the correct file permissions (chmod 600 authorized_keys). This can be useful for managing a fleet of multiple robots using the same SSH key for authentication. If using the SSH config file shown above for multiple robots, it is also useful to set the following additional options to avoid spurious warnings when switching between robots:

Host digit
    HostName 10.10.1.1
    User root
    IdentityFile ~/.ssh/common_digit_ssh_key
    StrictHostKeyChecking no
    UserKnownHostsFile /dev/null
    LogLevel QUIET

Note

Contact Agility Robotics support if you lose your SSH key or default WiFi password.

Network Time Protocol¶

The robot gets a time synchronization signal exclusively through GPS. Other clients on the network can synchronize to the robot’s GPS-synced clock through an NTP server running on the robot computer.

Other devices on the network will have to configure an NTP client to listen to the network’s broadcast address (10.10.1.255), the specifics of which are OS- and software-dependent. For example, to set this up on a Ubuntu 18.04 system mounted in the payload bay, perform the following steps:

  1. Install the ntp client program. This is done by using the command: sudo apt install ntp

  2. Configure the ntp client. Edit the file /etc/ntp.conf and add anywhere the following line: broadcastclient

  3. Reboot the payload computer.

  4. Confirm functionality by running the command ntpq -p. This will print out a status of the clock locking. If the broadcast address “10.10.1.255” has an asterisk next to it, then the payload computer has time-locked to the robot computer. This locking process can take a few minutes to complete.

An example ntp.conf file that works on an Ubuntu 18.04 payload computer can be downloaded here ntp.conf

Note

The robot will receive an accurate clock from the GPS device when signal strength is sufficient. When GPS is not available, the robot’s clock (and therefore the NTP server) will gradually drift out of sync with the global definition of UTC time, which will be corrected the next time a GPS signal is obtained. This will accomplished by slewing the clock rate to make up the difference.

JSON API Basics¶

The JSON Message API is the primary means of communication with the robot’s control software. It can be used to command actions and query information from the robot.

Tip

View the source of the terminal example page for an example of how to connect to the JSON Message API using JavaScript in a web browser and send messages. For a more fully-featured interface for experimenting with the JSON Message API, use the Message Composer in conjunction with the Gamepad Interface.

Protocol Basics¶

The JSON API uses a message-oriented client-server architecture. A client opens a WebSocket connection to the server running on the robot or simulator and sends and receives individual JSON messages. The robot and simulator listen on port 8080 by default, though this can be changed with a Configuration value. The subprotocol json-v1-agility must be specified. Multiple clients can be connected at the same time, but some messages (such as actions) are only accepted from a single client at a time, using a privilege system to arbitrate.

Once a connection is established, clients can send messages at any time. When a client sends a message, the robot may respond in the following ways:

  • Many messages will result in a single immediate response. One example is get-robot-info, which will be replied to with a single robot-info response.

  • Some messages can result in multiple responses, which may come much later than the original message. One example is notify-objects.

  • Some messages have an effect but do not generate any response if successful. One example is remove-objects.

  • If the message is malformed or a precondition is not met, the robot will respond with an error message. In this case, the normal response to the message is not sent and the message does not have its normal effect. One example is action-move while the client does not have the change-action-command privilege.

  • In some cases, the robot replies with a warning message in addition to the normal response. In this case, the message will still have its normal effect, but it may have done something slightly different than expected. One example is an action-move command with a higher velocity than is supported, in which case the velocity is clamped to the maximum and a warning is sent.

In addition to responses, the robot may send unprompted messages to the client at any time. One example is a privileges message sent when another client steals a privilege.

Tip

Use reference numbers to match up incoming responses from the robot with the sent messages that prompted them.

Message Format¶

Messages have the following basic format:

[
  "message-type-name",
  {
    "field1": "value",
    "field2": "value"
  },
  1234
]

All messages sent to and from the robot consist of a two- or three-element JSON array object, where the first element is a string representing the type of message, the second element is a JSON object containing data associated with that specific message type, and the optional third element is an int chosen by the client as a reference. The data object must be present even if it would be empty, while the third element may be omitted.

Important

The following are considered input errors and result in the robot sending an error message back to the client that sent them:

  • WebSocket messages that do not parse into valid JSON

  • WebSocket messages that do not parse into a two-element array of a string and an object or a three-element array of a string, object, and integral number

  • Message type names that are not a case-sensitive match to a valid message type

  • Message field names that do not match a valid field name defined for the message type

  • Message field data with the wrong formatting for the field’s type

Reference Numbers¶

The third element in the JSON array is known as the message’s reference number. It is optional, meaning that a two-element array containing just the message type string and data object is still a valid message. If a reference number is provided on a message sent by a client, any responses to that message sent out from the robot will have the same reference number. This is meant to allow clients to correlate messages in complex scenarios. If the reference number is omitted by the client, the message from the robot will also omit the reference number, which also always occurs for robot messages that are not responses to client messages. Any signed 64-bit integer is allowed, following the formatting rules listed in this document for type int. Clients can use any method to derive reference numbers, such as a monotonic counter or hash of the sent message, and the robot does not require reference numbers to be unique.

In addition to helping clients associate outgoing messages with incoming responses, reference numbers can be used to make specific modifications to highly nested action commands using action command modifiers.

Message Format Examples¶

Try sending some of the following example messages to the simulated robot using the Simple Terminal Example page.

A request-privilege message requesting the the change-action-command privilege with the default priority, which must be held by a client before it can send actions to the robot:

["request-privilege", {"privilege": "change-action-command"}, 123]

The privileges response from the robot, confirming that the client now has the requested privilege. This response contains the same reference number that was sent with the client’s original request-privilege message. The client did not specify the priority level in its request, so the default value 0 was used:

["privileges", {"privileges": [
  {"privilege": "change-action-command", "has": true, "priority": 0}
]}, 123]

An action-move command with no reference number, specifying a forward velocity while turning to the left:

["action-move", {"velocity": {"axy": [0.5, 1, 0]}}]

An more complex nested action that has the robot walk in a square pattern and then sit down. Each message nested in the action-sequential container has a different reference number, which allows referencing individual nested actions within this command with action command modifiers:

["action-sequential", {"actions": [
  ["action-goto", {"target": {"xy": [0, 0]}}, 0],
  ["action-goto", {"target": {"xy": [0, 2]}}, 1],
  ["action-goto", {"target": {"xy": [2, 2]}}, 2],
  ["action-goto", {"target": {"xy": [2, 0]}}, 3],
  ["action-goto", {"target": {"xy": [0, 0]}}, 4],
  ["action-sit", {}, 5]
]}]
Message Documentation¶

The documentation for JSON messages is split into topic-specific pages, such as Action Commands and World State, and a complete list of all messages can be found in the Index. Each message has a description and a list of fields. Fields are listed with a type, a name, and optionally a default value. If a field name has an equals sign and a value after it (e.g. int fieldname = 0), that field has a default value that is used if it is omitted from a sent message. Some messages are marked as Output only, meaning they are sent out from the robot to the client but are not accepted as input.

Field Types¶

Each message field has a type that imposes restrictions on what values it accepts. Some of these are equivalent to the basic JSON types, while some impose additional restrictions.

type int¶

Int values are transmitted as JSON numbers, and will accept any integer value within the range of a signed 64-bit two’s complement integer. Specific fields may be further restricted to values within a specific range.

Tip

Some JSON parsers (including JavaScript’s JSON.parse()) interpret all numbers as double precision floating point numbers, which can only losslessly store integer magnitudes up to 253. Larger integer values will lose precision, which can cause interoperability issues in some cases.

type float¶

Float values are transmitted as JSON numbers and are interpreted as double-precision IEEE-754 floating point numbers. Infinite and NaN values are not accepted. Specific fields may be further restricted to values within a specific range.

type string¶

String fields accept double-quoted JSON strings. Valid characters are printable ASCII characters within the range of code values from 0x20 (space) to 0x7e (tilde) and newline characters (0x0a). Standard JSON escape sequences are supported but only for escaped values within this range. Other characters, including all UTF-8 multibyte sequences, are silently dropped but do not cause the string to be rejected.

type bool¶

Bool values are transmitted as JSON booleans, and can have the case-sensitive literal values true and false.

type array¶

Array values are transmitted as JSON arrays, but each element must have a uniform specified type as defined in this section. Array fields can have further restrictions on their size, either by having a fixed accepted size or by only accepting sizes within a range. Array fields are denoted here as array<type, size> for fixed-length arrays and array<type> for variable-length arrays, though the latter may have length restrictions as noted in the field description. Array types can be nested, as in array<array<int>>.

type struct¶

Struct values are JSON objects similar to the second element of a complete message. That is, they only allow a specific set of field names and types. Structs can also be embedded in arrays or other structs. Each unique type of struct is given a name that appears in this documentation as a field type.

type enum¶

Enum values transmitted as JSON strings, but will only accept a limited set of case-sensitive values. Each unique enumeration is given a name that appears in this documentation as a field type.

type optional¶

Optional fields are denoted here as optional<type> and indicate that the field will accept the value null in addition to the values otherwise valid for the non-optional version of the field type. When an optional field is not specified in a message, it is treated as if null was sent.

type exclusive¶

Exclusive fields are similar to optional fields, except that no more than one of the fields marked with exclusive<type> can be specified and set to a non-null value within a given message or struct. Messages may consist entirely of exclusive fields, may have no exclusive fields, or may contain both exclusive and non-exclusive fields. It is generally possible to specify none of the exclusive fields in a message, but some message types may require that exactly one is specified.

type nested-message¶

Nested message values use the same formatting as a standalone message. Some nested message fields have restrictions on the types of messages they accept, which will be noted in the field description.

Note

These type definitions describe limitations on the formatting that the robot will accept, but the actual implementation may be observed to be more lenient in some cases. The implementation may change in the future to be more strict, so adhering to the guidelines listed here will help ensure compatibility with software updates.

Sessions¶

A new session is started each time a client connects to the robot, and records all of the state associated with that connection, such as privileges, action commands, and persistent queries. By default, sessions are transient, meaning that they are destroyed and forgotten once a client’s WebSocket connection closes. However, sessions can be made persistent, which allows them to survive disconnection and be resumed from another connection.

To convert a transient session to a persistent session, send the start-persistent-session message. Any existing state (privileges, actions, etc) from the transient session is retained. Persistent sessions can either be named, which use a user-specified name string to refer to the session, or anonymous, which use a randomly generated token sent back to the client to refer to the session.

To resume a disconnected persistent session, send the resume-persistent-session message from a connection associated with a transient session. The existing transient session will be shut down as if the connection had been closed, and all of the session state will be replaced with that of the persistent session. Persistent sessions can be resumed an indefinite number of times, but only one connection can be associated with a particular persistent session at a given time.

To shut down a disconnected persistent session or downgrade a connected persistent session to a transient session, send the stop-persistent-session message. Persistent sessions will also automatically shut down when disconnected for longer than the timeout period that was specified when the session was made persistent.

Note

While a persistent session is disconnected, any messages that would have been sent back to the client are queued up. When a client resumes the session, it receives all of these queued messages immediately. There is a limit to the number of messages that will be queued while disconnected, after which subsequent messages will be dropped (currently 1000).

Client Privileges¶

A privilege system is used to protect against accidentally sending potentially dangerous messages to the robot and to prevent multiple clients from interfering with each other. Privileges must be explicitly requested and are held exclusively by at most one client at any given time. Privileges are requested by sending request-privilege, and can be explicitly dropped by sending drop-privilege. The available privileges are listed under the privilege enum.

When a client requests a privilege already held by another client, it takes the privilege if the priority level sent with the request is equal to or greater than the other client’s priority, with zero being the default if left unspecified. When a client with a privilege drops the privilege or disconnects, the privilege is automatically given to another client that has requested the privilege, if any, with it going to the client with the highest priority level that requested it most recently. Whenever a client sends a request or its privileges change for any other reason, the robot sends out a privileges message.

message request-privilege¶

Requests a privilege. After receiving this command, the robot will send a privileges response listing the state of the client’s potential privileges.

privilege privilege¶

The name of the requested privilege.

int priority = 0¶

The priority level for exclusive privileges, capped at 100. Higher priorities may be used for internal purposes, which can cause a client to lose an exclusive privilege even with the maximum allowed priority. All negative values are allowed.

message drop-privilege¶

Drops a privilege. After receiving this command, the robot will send a privileges response listing the state of the client’s potential privileges.

privilege privilege¶

The name of the privilege to drop.

message get-privileges¶

Prompts the robot to send a privileges response. Note that this message cannot be used with query-group.

Common Structs¶

This section documents some common struct types that are used in many different messages.

struct pose¶

Represents the transform from a reference frame to a target frame. All of the fields of this struct are exclusive, each providing a different way of specifying the transform. Some of the fields do not fully specify a three-dimensional transformation; in these cases, the robot will choose appropriate values for the unspecified elements, which will generally be zero unless otherwise noted in the description for a message field. World State¶

The world state consists of what the robot knows about itself and the world around it. Using the JSON API, the world state can be queried to get data sensed by the robot and modified to inform the robot about things it cannot sense on its own.

Objects¶

Information in the world state is organized in terms of objects and their attributes. An object may represent a part of the robot, a box that the robot can manipulate, an image taken with one of the robot’s cameras, an abstract reference frame, and many other things. Attributes are specific types of data that can be associated with an object, such as its position, name, and shape.

Attributes¶

There is a predefined set of valid attributes that an object can have. Not all objects will have each type of attribute, and each object can only have one copy of a given attribute type. Some attributes can be queried and modified using the JSON API, while some are internal to the robot’s control software. Some attributes do not contain explicit data, but act as tags that can be applied to objects.

enum object-attribute¶

Represents the data attributes that an object can have associated with it. All of the fields of object-attributes correspond to values of this enum, but there are some additional attributes listed here that are not part of the struct.

option name¶

A name associated with an object. Objects do not have to have names, and object names do not have to be unique.

The ID of the parent object in the kinematic tree. Can be read but not modified with the object-attributes struct, but can be set using set-object-kinematics.

option children¶

The IDs of any child objects in the kinematic tree. Can be read but not modified with the object-attributes struct, but can be set using set-object-kinematics.

option pose¶

The pose of the object relative to its parent in the kinematic tree. Not present in the object-attributes struct, but can be accessed using set-object-kinematics and get-object-kinematics.

option velocity¶

The velocity of the object relative to its parent in the kinematic tree. Not present in the object-attributes struct, but can be accessed using set-object-kinematics and get-object-kinematics.

option timeout¶

A duration in seconds after which the object will automatically be removed. When any attribute is set, removed, or modified, the countdown resets.

option mass¶

The mass of the object.

option polygon¶

An array of [x, y] points that form the outline of the object’s flat polygonal geometry.

option april-tag-id¶

The AprilTags ID number associated with the object.

Tag indicating that this object is to be avoided by the navigation system.

option steppable¶

Tag indicating that this object represents a surface that the robot can step on.

option box-geometry¶

Selecting Objects¶

There are several different ways of identifying a particular object using the JSON API. Any message that requires specifying a particular object will use the object-selector field type, which is a struct that provides multiple options for specifying an object. The option that works for all objects is to use the object-id, which is a unique integer identifying each object. The other options used with object-selector fields are specialized to different types of objects.

While the object ID is guaranteed to uniquely identify a single object, other selection options may match multiple objects. In most cases, a single object is arbitrarily chosen when multiple objects match the selector. A few message types, such as notify-objects and remove-objects, instead act upon all of the matching objects. This distinction is denoted by the plural objects in the message names in place of the singular object.

Tip

To select objects based on more complex criteria than object-selector supports natively, use notify-objects with object-selector::has-attributes to narrow down the set of candidate objects, then use get-object to inspect each candidate. The chosen candidate object can then be specified by its object ID.

struct object-selector¶

Identifies one or more objects known to the robot’s control software. This is also frequently used to select the reference frame used to interpret a pose. In most cases, if multiple objects match the selection criteria, one of them will be chosen using the policy defined by the priority-order field. However, some messages (e.g. notify-objects) act upon all objects that match the selector.

This struct contains many optional fields, and multiple selection criteria can be specified at once. When multiple selection criteria are given, only the objects that match all of the given criteria are selected. Additionally, the include and exclude options allow for composing multiple selectors together.

Note

If none of the optional fields are provided, no object is selected, which generally results in an error. However, some message fields give a special meaning to an empty object-selector, which will be specified in the field description.

Modifying Objects¶

Objects can be added to the world state by JSON API clients. Other clients may query that object, and the object can be used internally by the robot’s control software, but the object cannot be modified or removed by any client except the one that added it. When a client’s session ends, all objects added by that client are removed from the world state.

Tip

Use persistent sessions to keep objects from being automatically removed upon disconnection.

Currently, the intended use case for user-defined objects is for use as reference frames. If an object is added and has its kinematic information set, it can be used as a reference frame for actions or queries. It is not recommended to set any attributes other than name, position, velocity, parent, or timeout at this time.

Caution

Because the world state is accessed by the robot’s control code, attributes should not be added to user-defined objects except in accordance with recommendations in this documentation or from Agility support.

message add-object¶

Creates a new object and optionally sets its initial attribute values. Responds with an object message containing the ID of the newly created object. Objects created in this way are associated with the session that created them, and can only be modified or removed by the client attached to that session. When a client disconnects and its session shuts down, all objects associated with that session are automatically removed. Action Commands¶

At any given time, the robot has an action command that defines its commanded behavior. This can be one of the atomic actions, or can be a composite action constructed using container actions. When an action message is sent by a client with privilege::change-action-command, it immediately replaces the current action command as long as it passes basic parsing and validation checks. The current action command can be queried by sending get-action-command.

When the robot starts up, its initial action command is action-idle. This action command is also applied whenever a client loses privilege::change-action-command. All commands that modify or query the action command require privilege::change-action-command.

The action command can be modified in-place by sending action command modifier messages. Changing the action command in this way allows an action nested within a container action to be modified without resetting the execution state.

Warning

Care must be taken to ensure safety when sending action commands in persistent sessions, especially with long timeouts. When a connection with a transient session disconnects due to loss of network connectivity or the client program crashing, any active action commands are immediately discarded and the robot goes idle, but action commands associated with a persistent session will continue to execute despite loss of direct control until the session is explicitly stopped, the timeout is reached, a client reconnects to the session and sends a new command, or the change-action-command privilege is taken away.

Execution State¶

The action command is always associated with an action status that summarizes the robot’s progress towards its commanded goal. Whenever the top-level action command’s status changes, an action-status-changed message is automatically sent to the client that commanded the action. This includes when the action is first commanded and when it is no longer the commanded action, in which case the reported previous or new status is action-status::inactive.

When an action command uses container actions to compose multiple atomic actions together, there can be a branching tree of active actions that each have a status. This tree is referred to as the execution state, and can be queried by sending a get-execution-state message. Some actions within the action command may be inactive, such as actions in an action-sequential container that have already finished running or have not been reached yet, and these actions do not appear in the execution state.

The leaf actions are the atomic actions in the execution state. While empty container actions can technically appear as leaves in the execution state tree, they are treated as if they contained an implicit action-idle inside them.

The execution state is completely reset when a new action command is applied. By using action command modifier messages to change parts of a complex nested action in-place, the execution state is preserved to the extent possible.

Compatibility¶

Multiple actions can be commanded to execute concurrently using the action-concurrent container action. If, as a result of this, multiple contradictory actions end up in the set of leaf actions at the same time, all of the leaf actions have their status set to action-status::failure and the robot performs the idle behavior instead. Contradictory actions are currently defined by associating all actions with a set of end effectors that they make use of, and if multiple actions that use the same end effector are executed at the same time, they are considered incompatible. Compatibility information is included in the description for each action.

One special case for compatibility is action-idle. It is always considered compatible with other actions, and any number of idle actions can be in the set of leaf actions at once. Idle actions are ignored if any other leaf actions are present. Because of this, idle actions can be useful as placeholders in complex nested actions.

Operation Modes¶

The robot has several operation modes that affect its behavior. Some actions are used to transition between modes, and some actions can only be used in specific modes. When a robot is commanded to execute an action that is incompatible with the robot’s current operation mode, that action’s status is set to action-status::failure and the robot performs the action-idle behavior instead.

enum operation-mode¶

Represents the current operation mode of the robot, which affects the behavior of its motors and the allowed actions.

option disabled¶

In this mode, the robot’s motors are not allowed to apply torque. The robot goes into disabled mode whenever the emergency stop button is pressed, or it can be commanded explicitly using action-set-operation-mode. When the robot is first turned on, it starts in this mode.

option damping¶

In this mode, the robot’s motors apply a fixed amount of damping, which causes the robot to fall in a semi-controlled manner when unsupported. This mode is automatically entered in a number of fault conditions, and can be commanded explicitly using action-set-operation-mode.

option joint¶

In this mode, the robot’s end effectors can be controlled while the base is assumed to be statically supported. This mode is used for diagnostics when the base is externally supported, and can be commanded explicitly using action-set-operation-mode.

option locomotion¶

In this mode, the robot’s legs are used for balance and locomoting around in the world, this is the normal operation mode of the robot. It can only be entered from the disabled mode by sending action-start.

option low-level-api¶

In this mode, the robot is controlled by torque commands given through the Low-level API. This mode can only be entered if there is a Low-level API client connected and sending commands (see Low-Level API).

message action-calibrate-imu¶

Commands the robot to move torso through a pre-defined set of poses to estimate the IMU’s compensation values, which include accelerometer bias and magnetometer hard/soft iron disturbances. Accelerometer bias introduces error into the robot’s pitch and roll state estimate, and can cause drifting while walking and general instability. Accelerometer bias will drift over time, and so should be re-calibrated every few months. Magnetometer hard/soft iron disturbances are introduced by stray magnetic fields and ferrous materials around the IMU and can interfere with the robot’s heading estimate. Note that during this action the robot should be standing on solid level ground and have room to move the base while keeping the feet stationary. Do not touch or move the robot while the action is taking place, as this will cause the calibration to fail.

The following conf values can be adjusted to modify the performance of this action:

  • action-calibrate-imu.enable-hsi-cal: The HSI calibration is disabled by default. To enable, set this conf value to true when running action-calibrate-imu

  • action-calibrate-imu.magnetic-north: This value should be changed to reflect the local magnetic north vector in Gauss. You can find the magnetic field model for your location here. The default value is [0.1984, -0.05295, -0.4733].

Mobility Parameters¶

Several parameters affecting locomotion are grouped into a common mobility-parameters struct. This struct is included as an optional field in several different actions. Default values for these parameters are used for any individual parameter that is not explicitly specified in an action. The default values can be read and modified using the default-mobility-parameters message.

struct mobility-parameters

All fields of this struct are optional. When sending mobility-parameters as part of an action or a default-mobility-parameters message, any fields not explicitly set are filled in by the current default values.

optional<array<float>> velocity-max¶

Sets the maximum absolute velocity allowed while moving. This velocity is defined in SE(2) so has the following form [v_x, v_y, v_rz] and units [m/s, m/s, rad/s].

optional<float> leg-length¶

Sets the nominal leg length in meters from the base origin to the ground while standing or moving. Decreasing this causes the robot to crouch further down, and increasing it causes the robot to stand more upright.

Caution

The default leg-length value was chosen to optimize power efficiency and stability, so only change it if necessary.

optional<float> step-clearance¶

Sets how high in meters the robot picks its feet off the ground when stepping. This can be increased to avoid tripping or stubbing on uneven ground, but this tends to make stepping louder as well.

optional<bool> avoid-obstacles¶

If this is set to false, the robot will ignore all obstacles while navigating to a goal or following a commanded velocity.

Caution

Disabling avoid-obstacles is not advised and can lead to unsafe behavior. Only expert users that have a good reason to disable avoid-obstacles should do so.

Note

The robot will not step onto terrain within obstacles for safety. If avoid-obstacles is disabled to force the robot through occupied areas, do not expect the robot to step onto terrain in those occupied regions.

optional<float> obstacle-threshold¶

If avoid-obstacles is enabled, the robot will attempt to stay this distance in meters away from obstacles while this action is active. If set to zero, the robot will still try to avoid colliding with obstacles, but will pass very close to them.

optional<bool> feet-avoid-unsteppable-regions¶

If this is set to true, the robot will attempt to avoid placing its feet in unsteppable regions of the terrain like edges of curbs.

Tip

If you find that locomotion performance is degraded in unstructured outdoor environments, try disabling feet-avoid-unsteppable-regions. This allows the robot to step on terrain that it might classify as unsteppable like tall grass.

Note

If you notice the robot avoiding stepping on parts of the terrain that you think should be fine to step on, it might mean your robot’s depth sensors are out of calibration. If you believe this is the case, please disable feet-avoid-unsteppable-regions and contact Agility Robotics support.

optional<bool> feet-match-terrain¶

If this is set to true, the robot will match the terrain height and slope with its feet while stepping. In addition, the feet may swing higher than normal to avoid colliding with the terrain while walking.

Container Actions¶

Container actions wrap one or more action messages (including both atomic and other container actions) and can be used to create compound actions or modify the how the status of an action behaves. They generally show up as non-leaf nodes in the execution state tree. All of these messages require privilege::change-action-command and can be used in the action command.

message action-sequential¶

Executes a set of actions in sequence. Tracks the currently executing child action and advances to the next when the current child is successful. Sets action-status::success when the last child in the sequence is successful, and sets action-status::failure when the current child fails.

array<nested-message> actions¶

The actions to execute in sequence. This array can be empty, in which case the action sets action-status::success immediately.

message action-concurrent¶

Executes a set of actions concurrently. The actions executed must be compatible. Sets action-status::success if all of its children are successful, and sets action-status::failure if any of its children fail.

array<nested-message> actions¶

The actions to execute concurrently. This array can be empty, in which case the action sets action-status::success immediately.

message action-timeout¶

Wraps a single child action, passing through its status until the child has been executing for longer than the timeout, at which point it sets action-status::failure.

nested-message action¶

The child action to execute.

float timeout = 0.0¶

The time limit for the child after which its status is overridden with action-status::failure.

message action-duration¶

Wraps a single child action, keeping the status at action-status::running before the duration is reached even if the child is successful, and then setting action-status::success once the duration is reached. Sets action-status::failure if the child fails regardless of the time. The duration can optionally start when the child action first succeeds, which is particularly useful for adding pauses in action-sequential containers without transitioning to the idle behavior in between actions.

nested-message action¶

The child action to execute.

float duration = 0.0¶

The duration to stay in the running state for before setting action-status::success.

bool after-success = false¶

If true, the duration timer starts when the child first sets action-status::success. If false, the duration timer starts when this action and its child first start executing.

message action-loop¶

Wraps a single child action, repeatedly re-executing it each time the child succeeds. Sets action-status::failure if the child ever fails, otherwise the status is kept at action-status::running. If an optional loop count is provided, the loop stops re-executing the child and sets action-status::success once the child reaches success after the given number of iterations.

nested-message action¶

The child action to execute.

optional<int> count = null¶

The number of times to re-execute the child action before the container is allowed to set action-status::success. If not provided, the container will loop forever.

Mapping¶

Digit can operate more effectively at certain tasks when it has a map of the physical space it is operating within.

Local Map¶

While in operation-mode::locomotion, Digit will use its sensor suite to generate a live-updating map of its local environment. This local map consists of two parts: a terrain map and an obstacle map.

Terrain Map

Digit’s estimates for the terrain height height in its immediate vicinity are stored in the terrain map, which is shown in the visualizer as a colored mesh. This terrain information is currently not used by Digit, but will enable Digit to navigate through more complicated terrain in the future.

Obstacle Map

Digit’s estimates for the presence of obstacles in its vicinity are stored in the obstacle map, which is shown in the visualizer as a black two-dimensional grid. Digit considers walls, people, and any tall object as an obstacle. The obstacle map is used by Digit to navigate through its environment while avoiding obstacles.

Caution

Although the obstacle map is Digit’s best estimate of obstacles in its vicinity, there is no guarantee that an obstacle will be detected. Certain environmental conditions or even material properties of an object can affect Digit’s ability to detect the presence of an obstacle. Caution is always recommended when piloting and moving near the robot.

Floorplan Maps¶

Floorplan maps consist of two-dimensional images showing the free and occupied parts of a physical space, such as the floorplan of the interior of a building. Floorplan maps are set by the operator of the robot and used by the robot for navigation and obstacle avoidance. The origin frame of a floorplan map can also be used as a reference coordinate system for queries and commands.

Sending a set-floorplan-map message enables the robot to use the map during the current session, while a clear-floorplan-map session removes it. Maps that are added are not used for navigation until the robot determines its position within the map, a process called localization. Localization can be done by setting the set-floorplan-map::initial-pose field, sending a set-floorplan-map-to-base-pose message, or having the robot observe landmarks that are present in the map.

Note

The robot can have multiple floorplans configured and active at once. Localization happens separately and independently for each map. This can be useful for avoiding performance issues while working in very large mapped spaces.

Note

After the set-floorplan-map message is sent, the map will not show up in the visualizer until initial localization has occurred. The easiest way to do this is in simulation is to set the set-floorplan-map::initial-pose field.

Landmarks¶

Landmarks are AprilTags that are present in the physical environment and are explicitly included in the floorplan map data sent to the robot. Landmarks in the floorplan data may have their location within the map explicitly specified, or they can be noted as included in the map but with their location unspecified.

When the robot observes a tag with an ID matching a landmark with a known position, it updates its estimate of where the map’s origin is relative to the robot’s state estimator origin frame. This allows the robot to perform both initial localization within a map as well as correct for drift or minor map inaccuracies. Landmarks with an unspecified position have their positions updated when the robot observes them, as long as the robot has already performed an initial localization within that map.

Landmark position estimates are continually updated by an internal filter, and the updated landmark positions can be extracted from the robot by sending a get-landmarks message. Landmarks can be added to a map after it is first sent to the robot using the add-landmarks message.

By default, the map origin frame is at the center of the image in map-data, with its X axis pointing up, Y axis pointing to the left, and Z axis pointing out of the screen. This pose applies an offset from the default origin frame to the new map origin frame. Only the planar (yaw, x, y) components are used.

If this pose is specified, the map starts out already localized such that this is the robot’s base pose relative to the map’s origin frame. Setting this field is equivalent to sending set-map without the initial pose and then sending set-floorplan-map-to-base-pose to localize.

Simulator¶

The Agility Robotics control program is able to run as a standalone simulator. It is distributed as a compiled AMD64 Linux AppImage. It primarily targets Ubuntu 18.04, but should work on most other recent Linux distributions. It has also been tested on Windows 10 using WSL. The simulator does not run on MacOS without a virtual machine.

Tip

See the Getting Started PDF distributed with the simulator for instructions on how to get it running on your computer.

Simulation Modes¶

The simulator can be started in one of two primary modes that affect how the robot’s movement is simulated.

Kinematic¶

Kinematic simulator mode is the default mode, and provides a fast, low-fidelity simulation of the robot’s movement. It assumes that the robot is able to move its body and joints exactly as commanded. It does not simulate collisions, inertia, or friction constraints. For simple tasks such as walking over flat ground and manipulating lightweight objects, the simulated robot will behave similarly to the physical hardware, but when the robot is pushed closer to its physical limits it will diverge. The kinematic simulator is sufficient for teleoperation training and testing user software interactions with the JSON API Basics.

Dynamic¶

Dynamic simulator mode simulates the inertial dynamics of the robot, along with collisions, friction constraints, sensor fidelity, and communication latency. As a result, the robot behaves much more similarly to the physical hardware in this mode than it does in the kinematic simulator mode, and this difference is particularly pronounced when the robot is performing more difficult tasks. Dynamic mode is not enabled by default and must be selected by changing a Configuration setting.

The dynamic simulator uses the MuJoCo physics engine to simulate dynamics and collisions. As a consequence, it requires a license and some extra setup to use. To run the dynamic simulator, the following files must exist in the user’s home directory:

  • $HOME/.mujoco/mjkey.txt

  • $HOME/.mujoco/mujoco200_linux/bin/libmujoco200nogl.so

Where mjkey.txt is the user’s valid MuJoCo license key file and libmujoco200nogl.so is from the MuJoCo 2.0 Linux archive downloaded from the MuJoCo website and extracted into the $HOME/.mujoco/ directory. If these files are in place, accessible to the simulator, and have the correct permissions, the simulator will be able to be started in dynamic mode.

Note

The dynamic simulator is slower than the kinematic simulator and may run slower than real time on low-end or old computers.

Note

When using the dynamic simulator mode on Windows with WSL, the Linux version of MuJoCo should be used instead of the native Windows version.

Simulation Environments¶

Custom simulation environments can be built by defining them in the Configuration, usually as an independent TOML file passed as an argument when launching the simulator. Objects are organized into models, which are kinematic trees of one or more primitive rigid body shapes, and one or more models (or multiple instances of a single model) can be included in the simulation environment by including them in the model list.

To get started with a simple example, save the following content in a file named world.toml in the same directory as ar-control and start the simulator with the example scene by running ./ar-control world.toml.

Example scene with the robot and a single box¶
# Model definitions
[world.package.body]
type = "box"
size = [0.4, 0.4, 0.4]
pose = {xyz = [0, 0, 0.2]}
mass = 1.0

# Models in the scene
[[model-list]]
model = "package"
pose = {xy = [1, 0]}

# The robot is implicitly included in the scene standing at xy = [0, 0]
Defining Models¶

A model is a named kinematic tree built up from one or more primitive rigid body shapes. In the example above, package is a defined model with a single body named body. There is always a predefined model named robot that is used to refer to the simulated robot.

Each model is defined by a table under the world table in the Configuration, and each body is defined by another table under the model table. The body table must contain the type and size of the primitive, and may contain a parent body name, pose offset, and body mass. See the example below and the complete reference for details.

Adding Models to the Simulation¶

Defining a model does not automatically add it to the simulated scene. To do so, a corresponding TOML table must be added to the model-list list of tables. Adding a table to a list of tables in TOML looks like the definition of a table, but with an extra set of square brackets around the table name.

The table that adds a model to the model list must contain the name of a model that exists in the configuration under the world table, and can optionally contain a pose offset for the model, a unique ID for the instantiated model, and can be made fixed instead of movable relative to the world. A single model can be included in multiple places in the simulation at different poses. See the example below and the complete reference for details.

Note

An infinite ground plane at z = 0 is always included in the simulation.

Important

In dynamic mode, non-fixed models should always be instantiated with initial poses that keep all of their constituent bodies outside of the floor or other bodies.

Important

The robot is included in the model-list even if not explicitly specified. It can be explicitly included in the list to move its starting position, but it should be 1 meter above the floor or a model surface to start up correctly in a standing position.

Complete reference example¶

# Complete model reference
########################################################################

# Each body in a model is a separate table under the 'world' table in the
# configuration. One body in each model must be the root body, which does
# not have a parent body.
[world.example-model.body-a]

# The name of another body in the same model that is the parent of the
# current body. The pose of this body is defined relative to the parent.
# The root body in the model must either have no defined parent or the
# parent must be an empty string.
parent = ""

# The type of primitive shape used for this body. Can be one of "box",
# "sphere", "cylinder", "capsule" or "none" if there is no primitive shape
# associated with this body (e.g. because it represents a global fiducial
# landmark). Cylinders and capsules have their major axis oriented along the
# z direction and all primitives have their origin at the geometric center.
type = "box"

# An array of numbers (or a single number) specifying the dimensions of the
# body in meters. The number and meaning of the elements depends on the
# primitive type as follows:
#   box:      size = [x, y, z]
#   sphere:   size = radius
#   cylinder: size = [radius, length]
#   capsule:  size = [radius, length]
size = [1, 0.9, 0.2]

# The pose of the body relative to the parent body's origin. Poses can be
# defined in the same ways that are allowed when specifying poses in the
# JSON Message API. This example uses roll, pitch, and yaw angles in degrees
# followed by x, y, and z translations in meters. The root body can have a
# defined pose, which defines an offset between the root body origin and
# the model origin. Defaults to the identity transformation if not provided.
pose = {xyz = [0, 0, 0.1]}

# The mass of this object in kilograms. Defaults to zero if not provided.
# Must be nonzero if the object is instantiated without rigidly fixing it
# in place by setting fixed to true in the model list. Does nothing in
# kinematic simulation mode.
mass = 3

# Defines the friction parameters of non-fixed bodies. The first element is
# sliding friction, the second is torsional friction, and the third is
# rolling friction. These values are passed directly to MuJoCo, so consult
# the MuJoCo documentation for guidance on setting these values. The
# default values of [1, 0.005, 0.0001] are usually sufficient, so friction
# usually does not need to be explicitly specified for a body. Does nothing
# in kinematic simulation mode.
friction = [0.5, 0.005, 0.0001]

# The color this body will be rendered as. The color can be a string
# containing any of the 140 standard HTML color names (case insensitive)
# or it can be a string containing a hex formatted HTML color preceded by
# the '#' character (e.g. "#FF00A1"). This value can also be set to "random"
# and the color will be randomly generated. Defaults to "random".
color = "MediumAquaMarine"

# Selects a user-provided mesh to use when visualizing this body. The mesh
# must use the GLB (GLTF 2.0) format. DRACO compression is supported. The
# value here must correspond to a file under one of the directories provided
# in the visualization.user-mesh-directories configuration option. If not
# provided, the body's collision geometry is visualized instead. When using
# mesh visualization, the color option is ignored.
#mesh = "example.glb" # Commented out so this works as a standalone example

# Rigidly attaches simulated AprilTags to the body. The tag poses are
# relative to the body origin. When the tag is within view of the simulated
# robot, the control software will act as if it sees a tag in the
# appropriate location. This can be used for both fixed and free models,
# and can be used to simulate localization within a premapped area. See
# the documentation for the set-floorplan-map message for additional details.
april-tags = [{id = 3, pose = {xyz = [0.5, 0, 0]}},
              {id = 4, pose = {xyz = [0, 0, 0.1]}}]

# This defines a second body in the model rigidly attached to the first.
[world.example-model.body-b]
parent = "body-a"
type = "box"
size = [1, 0.3, 0.2]
pose = {xyz = [0, 0, 0.2]}
mass = 1

# Creates a 1 degree-of-freedom revolute joint between this joint and its
# parent (if it has no parent this will override the default 6 degree-of-freedom
# joint between the root body and the world). Currently the only supported
# joint type is "revolute". Currently only joint can be defined per body.
# Adding joints currently has no effect in kinematic simulation mode.
joint.type = "revolute"

# The position where this joint is attached, relative to the parent body's
# origin.
joint.offset = [0, 0, 0]

# The joint limits (in degrees for revolute joints). By default joints
# have no limits.
joint.limits = [0, 180]

# Specifies the axis of rotation for revolute joints. Defaults to
# [0, 0, 1] if not provided.
joint.axis = [0, 0, 1]

# Coulomb friction
joint.damping = 0.1

# Dry friction
joint.friction = 0.1

Gamepad Interface¶

The gamepad interface is designed for the gamepad supplied with the robot, but can also run on standard desktop/laptop computers. Multiple users can have the gamepad interface connected to the same robot or simulator at once, but only one user can send commands at any given time.

Tip

If the “Command Privilege” button is not green, the gamepad interface does not have the change-action-command privilege, and none of the buttons in the interface that send commands to the robot will work. Press the button to request the privilege.

Layout¶
Menu

This button toggles the interface menu in the upper left corner fo the screen. This menu shows a collapsible list of objects in the visualizer as well as some auxiliary robot information. Some of these objects have parameters that can be changed through the menu, such as object visibility, colors, lighting, etc.

Console

This button toggles the console display in the upper left corner of the screen. This console will display the latest event/warning/error messages received from the robot in order received. When closed, the console button will change color to notify the user of new messages (yellow for warnings and red for errors), it will only return to normal color if the console is opened.

Help

This button toggles the interface help in the upper left corner of the screen. This information shows the button mapping for the current mode and will automatically update when the mode is changed.

Name

Shows the name of the robot in the upper center of the screen.

Time

Shows the current running time of the simulation or robot. When connected to the simulator, this represents the simulation time, which might increase faster or slower than real time depending on the computer speed and whether free-running is enabled in the Configuration.

Battery

Shows the current battery charge of the robot. Always displays “100%” when connected to the simulator.

Mode

Displays the current operation mode of the robot. Pressing this button cycles between locomotion mode, damping mode, and disabled mode, commanding the robot to sit down and stand up if necessary.

Stand

This button puts the gamepad interface in stand mode, which allows the user to adjust the robot’s pose while standing in place. Other gestures can also be commanded such as waving or bowing.

Move

This button puts the gamepad interface in move mode, which allows the user to command the robot to walk in a specified direction. Other properties of the move behavior such as obstacle avoidance and obstacle threshold can also be adjusted.

Goto

This button puts the gamepad interface in goto mode, which allows allows the user to command the robot to walk to specified waypoints. Other properties of the goto behavior such as obstacle avoidance and obstacle threshold can also be adjusted which will impact the planned path.

Manipulation

This button puts the gamepad interface in manipulation mode, which allows the user to command the robot to pick and place objects. Objects can either be selected in the scene to pick or place on top of or if no object is selected it will default to the closest one.

Command Privilege

Pressing this button requests or drops the change-action-command privilege, which allows the gamepad interface to send commands to the robot. The button will be green if this instance of the interface has the privilege, but will automatically turn gray if another JSON API client takes the privilege.

Logging¶

A variety of operational data is captured through the logging system and are enabled/disabled via configuration. Various modules log to separate files but their locations depend on the operational mode. When running on hardware each log session is saved in the same timestamped folder. When running in simulation, each log session is overwritten by the current session, excluding the summary log.

Summary Log

The summary log records a brief, human-readable summary of various vital statistics for the robot each time it is powered on, including active time, temperatures, battery charge, power usage, and computer performance. These logs are never automatically cleared to save space, and are used to track the robot’s life cycle for scheduled maintenance events and to detect potential problems before they cause an issue. Summary logs can be downloaded from the robot’s web interface, and you may be asked to provide these by customer support to help diagnose a problem.

Event Log

The event log provides a per session, human readable overview of important events logged by the system. Events may include software events (action failures, bugs, etc.), hardware events (sensor diagnostics, temperatures, voltages, etc.), as well as errors, warnings, and failures during operation. Event logs can be easily viewed through the Event Viewer.

Visualizer Log

The visualizer log captures a detailed record of what was visualized during a session. These logs may be played back through the Visualizer Playback Tool for a variety of purposes, and are useful when conveying robot performance during a session.

Cyclic Log

The cyclic logs capture a much more detailed record of events when running on hardware. These capture time series information and are continuously written to in a fail safe way.

Tip

The Summary Log, Event Log, and Visualizer Log are useful when attempting to diagnose robot issues, and are a useful tool for the support team. The Cyclic Log is much more detailed and not required for the initial service request.

Visualizer Playback Tool¶

The Visualizer Playback Tool provides an interface for a user to view visualizer log files. The interface is intended for a singular user.

Important

Configuration¶

When the control program is started, it reads key-value configuration information from a set of TOML files. Some of these configuration values can be changed by users.

Setting Configuration Values¶

There are several different ways for users to set configuration values when launching the control program.

TOML Files¶

Configuration values can be loaded by writing them in a TOML file (version 1.0 syntax is supported) and passing the file path as a command line argument to the control program. The filename must end in the .toml extension. Multiple files can be specified. The values in this file will override any default configuration values, and files appearing later on the command line will override values from earlier files if they both try to set the same value.

Example: ./ar-control settings.toml

Modes¶

Some preconfigured TOML configuration files used to enable different “modes” are bundled with the simulator. These can be activated by specifying the name of the mode on the command line.

Example: ./ar-control low-bandwidth

low-bandwidth

Alters some configuration settings to optimize the simulator or robot for teleoperation over a low-bandwidth and high-latency communication link, such as a cellular connection. The visualizer will update less smoothly and may omit data-intensive items like terrain and obstacle maps, and some interactions with the JSON API may be rate-limited.

visualizer-playback

Allows the user to playback visualizer log files using the Visualizer Playback Tool . Playback is only available in this mode and is not intended to be run on hardware.

Command Line Arguments¶

Individual configuration values can be specified directly on the command line. Any values specified in this way will override default values and values in any TOML files specified on the command line.

Example: ./ar-control 'simulator.dynamic = true'

Note

A key-value pair must be provided as a single command line argument, which means spaces and certain characters need to be escaped or quoted appropriately for the shell you are using. The simplest way to do this in a bash-like shell is to wrap the configuration key-value pair in single quotes, as shown above.

Configuration Values on Hardware¶

Note

Configuration values should not need to be changed often on hardware. Before following any instructions in this section, make sure your application requires configuration changes.

Important

Do not modify or overwrite the 99-tuning.toml file in the /agility/conf folder unless directed by Agility Support. This file contains robot-specific tuning parameters and will cause reduced performance if modified.

To change hardware configuration values, TOML files with the desired parameters should be copied to the robot using the Configuration Browser at http://10.10.1.1/configuration/. The files will be loaded by the robot in lexicographical order (e.g. a file named 99-config.toml will override any common values specified in 00-config.toml).

Invalid Overrides¶

When an invalid configuration override is specified by a user through any mechanism, an error message is printed to the terminal and sent to all JSON API clients. Additionally, that invalid override is not applied to the robot’s active configuration. An invalid configuration override can either mean trying to set a value that doesn’t exist, or setting an existing value with the wrong type of value. The same types of error messages are also printed and sent when a nonexistent mode is specified.

Common Configuration Values¶

The following is a list of the more commonly used configuration keys, along with their default values when running in simulation.

Important

Always double-check the spelling of configuration keys. Currently, no error messages are given for misspelled or nonexistent keys. Configuration keys are case-sensitive.

Note

Default values given are for simulation. These values may vary on hardware.

simulator.dynamic = false

If this value is false, the simulator is run in kinematic mode. If this value is true, the software will try to run the simulator in dynamic mode, but will fail if it cannot find a working MuJoCo installation and license key.

simulator.free-run = false

If false, the simulator does not run faster than real-time. If true, the simulator will run as fast as the hardware allows. Applies to both the kinematic and dynamic simulator, though the kinematic simulator is capable of running much faster than the dynamic simulator on most hardware.

Sets the TCP port that the control program serves files and websocket connections from. Sets the number of ports to try when automatically selecting a port. Increase this to run more simulators in parallel with automatic port selection.

Advanced Topics¶

Low-Level API¶

The Low-level API is intended for advanced users who would like to write their own locomotion and manipulation controllers. The Low-level API enables this by exposing low-level sensor data and allowing users to give commands directly to the motor drivers. For a full description of the signals available to send and receive via the Low-level API, see the API source code and example files found here. The source code has been developed and tested on Ubuntu 18.04, but should work with most other recent Linux distributions.

Important

The low-level API will only function when used with hardware or the Dynamic simulator.

Caution

While the low-level API is active, all faults and warnings from the hardware will be ignored. Error messages should be monitored on the web interface and appropriate action taken by the operator.

Specifications¶

Caution

When developing custom controllers, they should be tested extensively in simulation before attempting to run them on the robot. Unstable controllers have the potential to cause serious damage to the hardware and injury to anyone nearby. For more information, see Safety.

The Low-level API streams data using the UDP protocol and a binary message format. It is not necessary to interface directly with the data streams or the message format, as the low_level_api .c and .h files provide a high-level interface for sending torque commands to the robot and receiving sensor data. low_level_api.h defines the observation (i.e. sensor data from the robot) and command (i.e. motor commands to the robot) structs used by the interface.

The lowlevelapi_example.c file provides a template for creating custom control code. When creating custom code, keep in mind that you are responsible for handling latency and timing on the payload computer. When a command is sent over UDP it will be applied to the motors within 1.5-2 ms, but any delay introduced on the payload side will be added to this amount. If you take advantage of the velocity and damping commands, these control loops are handled directly by the motor amplifiers, so the control loop stability will not be impacted by any transmission latency.

Recieving Data: The Low-level API will only send observations while there is an active connection to an external client. To receive sensor data, commands must be periodically sent to the Low-level API to keep the connection active. If you want to read sensor data without enabling torque control you can set the apply_command flag to false in the command struct.

Sending Commands: To enable torque control via the Low-level API, you must do the following:

  1. Send commands from a connected client with the apply_command flag set to true

  2. Once the client is connected and sending commands, transition to Low-level API operation mode with action-set-operation-mode using the message composer or custom websockets code

Caution

Once you change the operation mode, the robot will be fully controlled by the client connected to the Low-level API. Make sure that the client code can react according to the robot’s current state. If the robot is in the middle of another action and the client is unprepared to take over control, the results are unpredictable and could cause damage to the robot. We suggest implementing your own checks during this handoff to make sure that the robot commands are not too different than what the default controller is already applying. In the event of a large discrepancy, the apply_command flag can be set to false to hand control back to the default controller.

Disconnecting: The control code on the robot can be directed to resume control at any time by using action-set-operation-mode to transition to the desired operation mode. If the operation mode is set to locomotion, the robot will attempt to stand from whatever state it is currently in, taking recovery steps if necessary, or falling if a transition is impossible.

If the Low-level API stops receiving commands from a client (either because the client connection has timed out or the apply_command flag has been set to false), the control code on the robot will resume control and transition to the operation mode specified in fallback_opmode in the command struct. The fallback_opmode field should be set to an appropriate state at all times in case of unexpected disconnections. Note that options for fallback_opmode are limited to locomotion, damping and disabled. The connection timeout for clients can be set with the configuration value lowlevelapi.timeout.

While connected to a Low-level API client, the host can still be shutdown by commanding an action-shutdown via the host’s web interface or by briefly pressing the power button on the side of the robot.

Safeties¶

Before the torques, target velocities, and damping parameters are sent to the motor controllers they may be modified in the following ways (note these only take effect when the emergency stop is released).

  • If zero torque and zero damping are commanded the robot will override the damping parameters with a medium amount of damping.

  • If a joint is within 0.1 radians of the hard stop (defined in the kinematic tree) a PD controller will attempt to push the joint away from the hard stop. Additionally, the torque commanded by the user will be scaled down to zero linearly as the motor approaches the hard stop.

  • If excessive velocity commands are given to joints near hard stops, the velocity command will be reduced. This reduction is based on the dynamics of the motor, ensuring that the joint can come to a complete stop within 0.1 radians of the hard stop under maximum acceleration.

  • A 2 second ramp will be applied to the commanded torque whenever the emergency stop is released and a non zero torque value is received. The ramp is applied to torques generated by the Agility Robotics control program as well as torques received through the Low-level API but the ramp is only applied once per emergency stop release. For example, if the user stands the robot up and then takes control through the Low-level API, the torque ramp is only applied during the initial standing up of the robot and not applied separately to the user commanded torque.

  • Robot error information is available via the Low-level API observation. This flag will indicate when an issue occurs that causes the system to have reduced performance, or when a fatal issue may occur unless the user does something. When this flag is set to true the system will automatically disable the Low-level API after a brief delay. This delay can be read by calling the llapi_get_error_shutdown_delay() function from the API. It is highly recommended that users monitor this flag and have the custom controller move the robot into a standing pose when it is set to true. Specific fault information can be found by monitoring the JSON API or gamepad interface.

Caution

These safeties are intended to minimize the risk of damage from experimental controllers. However, the Low-level API provides direct control over motor torques and velocities and it is impossible to design safeties to allow all reasonable behaviors while prohibiting dangerous ones. You should not rely on these safeties to catch any bugs or errors in your controllers, and should instead implement your own safety code and experimental procedures to prevent failures from causing damage or injury.

Configuration¶

In the Low-level API example code there are several example files illustrating configuration values for common testing setups. To apply the configuration values in each file, see Configuration.

  • lowlevelapi_example.toml: Contains the configuration values directly related to the performance of the Low-level API. Use this file every time you run the Low-level API

  • lowlevelapi_example_teststand.toml: Defines the robot model as fixed to the world frame at 1.2 meters off the ground. Useful for testing joint level behaviors and inverse kinematics where the robot does not need to balance

  • lowlevelapi_example_rope.toml: Defines the robot model as starting 1 meter off the ground. Also defines a disturbance force simulating a rope holding the robot suspended. Useful for testing controllers that start with the robot hanging from a winch or gantry

  • lowlevelapi_example_ground.toml: Defines the model in a face-down lying pose, for testing code where the robot will stand up from the ground

Assuming a Ubuntu 18.04 system installed in the payload bay, perform the following steps to set up access to ROS data streams.

  1. Set up the payload ethernet network and NTP synchronization using the instructions in Networking.

  2. Install ROS Melodic. Good installation instructions can be found here.

    • export ROS_MASTER_URI=http://10.10.1.1:11311/

    • export ROS_IP="Payload computer IP address"

  4. Add an entry to /etc/hosts for your robot IP address and robot name. For example, add “10.10.1.1 digit0XX” where XX is your robot ID number.

  5. Confirm network connection to the robot. You should be able to ssh to the robot with the command “ssh root@digit0XX”. If a password is requested you can establish a shared key system by issuing the command “ssh-copy-id root@digit0XX”.

Changelog¶

  • Release 2021.06.01

    • Features

      • Low-Level API Improvements

        • Low-Level API now automatically disengages when robot faults are present for longer than a few seconds. See the LLAPI Safeties section for details.

        • Added additional information (motor velocity limits and robot fault status) to the Low-Level API to provide users more visibility into robot capabilities.

        • Added additional information about how safeties are implemented. See the LLAPI Safeties section for details.

      • Simulator Improvements

        • The simulator can automatically choose a port to run on in case of conflict between multiple instances by setting server.port = 0 in the configuration. See the server.port, server.port-autoselect-start, and server.port-autoselect-attempts configuration values for details.

        • The simulator now provides an error message when given a configuration value with an invalid name instead of silently ignoring it.

        • Simulator models can now have movable joints between bodies.

        • Log files can be enabled in the simulator and have their save directory set. See the logging.log-directory, logging.summary-log-directory, logging.enable-event-log, logging.enable-visualizer-log, and logging.enable-summary-log configuration values for details.

      • Added Mobility Parameters

        • Moved the speed, obstacle-threshold, and avoid-obstacles fields from action-goto and action-move into the mobility-parameters struct. These fields will still be accepted by the robot as deprecated alternatives to the mobility-parameters struct in this release, but support for them will be removed in the following release.

        • Added feet-match-terrain, feet-avoid-unsteppable-regions, leg-length and step-clearance parameters to the mobility-parameters struct.

        • Several actions (action-stand, action-move, action-goto, action-pick, action-place) can now accept a mobility-parameters struct that can selectively override the default parameters set with default-mobility-parameters.

        • Added controls to gamepad interface menu for dynamically modifying mobility-parameters.

        • The robot will attempt to avoid unsteppable regions such as the edges of curbs when the feet-avoid-unsteppable-regions parameter is enabled.

      • Added get-config-info message for querying information about the robot’s loaded configuration data. A web interface was also added to quickly view this config-info to help debug configuration related issues.

      • Additional documentation was added detailing the various types of logs available on the robot. See the Logging section for details.

      • Large log files are automatically split when they exceed a size limit.

      • Added an event log viewer to the hardware web interface that allows users to quickly load, filter, search, and view event logs.

      • Added visualizer-playback mode which provides an interface for loading and playing back visualizer log files.

      • Added low-bandwidth mode that improves responsiveness and visualizer performance over low-bandwidth and high-latency connections.

      • The limit on the number of user-added objects was increased from 1024 to 10000.

    • Bugfixes

      • Fixed a bug that could cause get-snapshot to return corrupt images.

      • Added additional safeties to resist knee four-bar inversion when commanding high torques and velocities through the Low-Level API.

      • Fixed bug that was delaying query responses while the simulator was paused.

      • Changed HTTP headers to allow custom HTML interfaces to reference scripts and files served by the simulator or robot.

    • Breaking Changes

      • In messages sent from the robot, pose, gps-pose, and spatial-vector structs will have no fields set (e.g. pose: {}) instead of using pose: {"rpyxyz": [0, 0, 0, 0, 0, 0]} if the pose or vector has none of its components specified. The rpyxyz option is still used in all other cases.

  • Release 2021.04.18

    • Features

      • The model-list array used to define simulation objects can now be split across multiple config files.

      • Added several new simulator specific messages (simulator-emergency-stop, simulator-start, simulator-pause, simulator-set-pose, simulator-apply-force).

      • Simulated objects can use user-provided meshes for visualization. See the visualization.user-mesh-directories configuration value for details.

      • The action-shutdown message now works in the dynamic simulator.

      • Selecting objects with object-selector in the JSON API now allows using multiple criteria to narrow down the set of results, as well several new selection options (name, owned, user-object, descendant-of, ancestor-of, include, exclude, priority-order).

      • action-status now has an associated information string that explains the reasons for failure, and sometimes provides additional context when an action is running or successful.

      • Sensor frames can be accessed using the JSON API.

      • The robot will attempt to match the terrain height and slope with its feet while walking.

      • Added a gaze sequence to the end of action-start which helps the IMU orientation settle.

      • action-calibrate-imu can now compensate for magnetometer hard/soft iron disturbances.

      • Improved feedback when updating software using the web interface.

      • Changed how selected objects are indicated in the gamepad interface.

      • Added documentation for action-start modes (start-mode::push-up, start-mode::rope, start-mode::assisted).

      • Improved documentation for IMU calibration and battery safety. See IMU Calibration and Battery Storage sections for details.

      • Improved navigation between the web interface pages.

    • Bugfixes

      • Fixed bug that sometimes caused objects to be initialized with the wrong orientation in the dynamic simulator.

      • Fixed bug where contact friction values weren’t properly applied in the dynamic simulator.

      • Fixed bug that would cause poor GPS tracking performance in some situations.

      • Fixed remove-action command.

      • The robot responds more quickly to sudden connection loss, which can happen if the gamepad runs out of power or if the network connection is unreliable.

    • Breaking Changes

      • Removed /robot_state topic from ROS, the recommended way to poll the current state of the robot is through the JSON API.

      • Objects with the timeout attribute have their removal countdown reset when any attribute is modified, not just the timeout attribute.

      • Removed explicit reference-number field from action-status-changed message. Use the reference number of the received message instead.
